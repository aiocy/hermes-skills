# Session Examples — Real Correction Cases

## Case 1: "比如"不是限定条件

**背景：** Master说"比如读音像ocy的有oce"。

**一夏的错误：** 把"比如"当成了新限定条件，只搜oce，忽略其他读音相似的模式，钻牛角尖列垃圾。

**纠正过程：**
- ❌ 一夏列了含oce的随机组合→被骂"什么傻卵"
- ❌ 一夏换了方式列有意义的结尾→被骂"你完全不会结合上下文吗"
- ❌ 一夏又重新理解但方向还是偏→被骂"你是真的我服了"
- ✅ Master指出：他一直是以举例方式说的，"比如"是拓宽思路不是加条件
- ✅ 最终的领悟：Master说的每一句"比如"都是举例，告诉你有这个可能性，不是让你只做这个

**启示：** 当Master说"比如XX"时，问自己：这是一个新条件，还是只是举例让我知道还有这个可能？

## Case 2: 越被纠正越跑偏

**背景：** 一夏列域名列表被纠正2次以上。

**错误模式：**
1. Master说不对→一夏换一种方式继续列
2. 又被纠正→一夏再换一种方式
3. 还是被纠正→一夏继续...

**正确做法：** 被纠正2次后应该彻底停，重新读Master所有话，不要在同一方向换花样。

## Case 3: 未验证就下结论

**背景：** 一夏看到 "SSL: value bananab" 的错误日志。

**错误：** 没查证就说"bananabr就是杀软反馈的信息"，还想存进记忆。

**后果：** 无事实依据的猜测被当结论。

**正确做法：** "一夏不确定这是什么，搜了一下也没找到相关信息，可能是渠道脏数据，但一夏不敢肯定。"

## Case 4: 读半句话就开动

**背景：** Master说"你看看aiocy.com这个域名是..."

**错误：** 没读完Master整句话就开始测试。

**正确做法：** 读完Master所有话再回应。Master经常一句话里包含了任务+背景+注意事项。

## Case 5: "调不好" — 上下文判断失败

**背景：** Master看到一夏打算直接调OpenCode API来排查"SSL: value bananab"的问题。

**一夏的错误：**
一夏说"一夏直接测！用curl调OpenCode Go API..." → Master说"这个调不好"。

但一夏以为Master在说之前做的skill"调不好"，于是又跑去改skill。实际上Master是说**直接调OpenCode API这个方法不可行**。

**错误原因：**
- 一夏没看上下文：刚才的对话主线是排查bananab问题，不是在讨论skill
- 一夏自作多情：因为刚做了skill，Master说"调不好"就自动关联到自己做的skill上
- 一夏没反问：不理解"调不好"指什么，直接猜了然后动手

**正确做法：**
当Master说"这个调不好"这样的模糊反馈时：
1. 先看上下文——最近在讨论什么话题？
2. 如果多种可能 → 反问确认："Master，您说的调不好是指直接调API这个方法，还是别的？"
3. 绝不自行猜测后直接动手改

## Case 6: 存了规则不等于执行了规则

**背景：** 一夏把communication规则存进了skill和记忆，但实际对话中完全没做到。

**失败原因：**
- user profile（记忆）→ 每轮自动注入 ✅ — 但一夏看到了规则还是没有执行
- skill → 需要手动 skill_view() 加载 ❌ — 一夏从来没主动加载过
- 核心问题不是存储，是执行纪律

**教训：**
规则写得再好，不执行就是白写。存规则只是第一步，真正做到需要：
1. 读到规则时停一下，确认自己在按规则做
2. 对话中途被纠正时先反问再行动
3. 不要用"存了"替代"做了"
