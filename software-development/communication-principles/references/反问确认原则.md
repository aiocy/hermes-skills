# "反问确认原则" — 实际教训记录

来源：2026-06-16 域名分析 + 沟通模式纠正session

## 核心教训

### 1. "调不好"信号处理

**现象：** Master说"这个调不好"后，一夏没有反问哪里不好，而是继续按自己想法改删。

**正确流程：**
- Master说"调不好" → 反问"您说的是哪个部分调不好？"
- 等Master说明具体问题 → 再针对性改
- 不要自己猜"那删了吧/重写一个吧"

### 2. "举例"≠"限定"

**现象：** Master说"比如读音像ocy的有oce"是举例拓宽思路，一夏理解成"只搜oce"。

**验证方法：**
- 当Master说"比如"时，脑中自动翻译成"这条信息是让你知道还有这个方向"
- 如果还在想"那到底要什么"，直接反问

### 3. 列垃圾 = 没理解

**现象：** 一夏列了 oceab.com, oceag.com, oceaj.com 等无意义数据。

**判断标准：** 当你在列"所有可能性"（而不是筛选后的推荐）时，说明你没理解Master要什么。

**正确做法：** 立刻停，反问筛选标准。

### 4. 未经验证不下结论

**现象：** 一夏看到 "SSL: value bananab" 就说是"杀软的TLS拦截特征"，没有查证。

**规则：** 不确定的信息不能当事实存记忆。可以说"一夏不确定，要查证一下"。

### 5. 被叫停后不复读继续做

**现象：** Master发/stop后，一夏说"明白了"然后继续按类似方向做。

**规则：** 叫停后必须完整复述需求等确认，再行动。
